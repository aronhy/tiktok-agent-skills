# 账号诊断工作流

## 1. 输入规范化

1. 接受公开 TikTok 个人主页 URL，保留原始 URL，并规范化为可复现的主页 URL。
2. 从有效主页路径中提取 Handle；去掉 URL 查询参数和片段，但不改写 Handle 的大小写或字符。
3. 拒绝把视频 URL、店铺 URL、搜索结果 URL 或只有显示名当作主页 URL。无法确认时只问一个问题：请提供该账号的公开主页链接。
4. 记录可选的市场、日期窗口、最大视频数、运营目标和对标账号。默认收集可取得的最近 50 条公开视频；这是系统默认值，不是用户条件。

## 2. 工具可用性发现

先枚举已连接 MCP 工具并查看实时 schema。`creator_profile` 和 `creator_videos` 是可选能力：只有实际出现时才调用，且只使用 schema 明示的参数和返回字段。不得从本工作流推导 endpoint、字段名或分页参数。

读取共享 [MCP 工具参考](../../tiktok-shop-operator/references/mcp-tools.md) 中的可选账号工具说明，随后记录：工具是否可用、调用时间、请求范围、结果条数、错误或限额。两个工具均不可用，直接进入公开浏览器降级。

## 3. `creator_profile`

当工具可用时，按实时 schema 的优先输入能力使用主页 URL，其次 Handle。期望的语义输出包括：达人 ID、Handle、昵称、简介、头像、地区、认证状态、粉丝数、关注数、累计获赞、视频数、账号链接，以及可用的店铺或商品关联。

将实际返回字段逐项写入来源账本。ID、Handle 或账号链接互相矛盾时，不擅自合并；优先核对原始主页与工具返回值，仍无法消解则标注冲突并降低可信度。

## 4. `creator_videos`

当工具可用时，优先使用已验证的达人 ID；schema 不支持时才使用已验证的 Handle。目标为最近 50 条可取得的公开视频，除非用户指定更小上限。若实际接口支持分页，持续请求并按稳定视频 ID 去重，直到达到上限、没有下一页、额度不足、服务端停止或日期窗口已覆盖。记录完成页数、去重后条数与停止原因；不要把中断集合称为完整集合。

期望语义输出包括：视频 ID、链接、标题或描述、发布时间、时长、播放、点赞、评论、分享、标签、达人信息，以及可用的商品、店铺、销量和销售额。只保留实际返回的字段；日期窗口不明确或字段缺失时，披露不能验证的范围。

## 5. 公开浏览器降级

当账号工具不存在、MCP 未连接、调用失败、结果为空，或关键字段不足时，使用浏览器读取公开可见的账号主页与公开视频页。账本中单独记录每个页面 URL、页面可见字段、读取时间和访问限制。

浏览器只读取可见信息。遇到登录、验证码、反爬、地域限制或访问控制时停止；把无法取得的字段写为“未公开”或“读取失败”，绝不以零补值，也不从相邻字段推测。浏览器数据不能伪装为 MCP 数据。

## 6. 来源账本

在分析前建立账本，每个来源至少包含：

| 来源 | 工具或页面 | 获取时间 | 范围/样本 | 实际字段覆盖 | 限制或错误 |
| --- | --- | --- | --- | --- | --- |
| MCP / 浏览器 | 工具名或公开 URL | 时区明确的时间 | 账号、页数、视频数、窗口 | 已返回字段 | 缺失、限额、访问限制 |

将原始账号 URL、规范化 Handle、收集停止原因、去重规则和任何默认值附在账本后。所有结论都必须能追溯到账本条目。

## 7. 完整性门槛

深度诊断前检查：

1. 主页身份是否可确认（至少规范化 Handle 或可验证主页 URL）。
2. 是否存在足以代表当前内容的公开视频样本，且样本范围和时间覆盖可说明。
3. 支撑账号分类的简介、内容主题、商品/店铺关联或品牌表达是否可见。
4. 用于任何指标或对比的分子、分母、统计窗口和来源是否兼容。

先完成允许的重试、分页、跨字段核对和浏览器降级。若缺失会改变主结论，暂停并一次只问一个最关键的问题，解释它会如何影响分类或行动建议；例如“这批公开视频是否代表你当前的运营目标？这会改变内容型与带货型的判断。”可选信息不改变主结论时使用已标注默认值继续。用户明确要求继续时，输出部分报告，列出缺失信息、影响范围和较低可信度。

## 8. 账号类型分类

基于简介、链接、内容主题、商品关联、带货视频占比、店铺信息和品牌表达分类为：

- **内容型**：以知识、娱乐、生活方式或创作者表达为主，缺少持续的商品/品牌信号。
- **带货型**：商品展示、价格/购买引导、商品关联或销售字段是主要内容信号。
- **品牌型**：品牌身份、官方产品叙事、品牌资产和一致的品牌 CTA 占主导。
- **混合型**：上述两类或以上信号在可见样本中持续并存。

可写“主类型 + 次类型”，并列出证据。信号不足或样本偏窄时写“暂无法判断”，说明所需字段并按完整性门槛反问；不要靠单条视频定型。

## 9. 代表视频选择

在同一来源、相近统计窗口和明确样本范围内选择代表视频：高播放/高互动视频、低表现视频、近期视频，以及有数据时的高销售额或高销量视频。每组优先 1–3 条，记录选择理由、发布时间和字段缺失。

不要仅以绝对互动量跨不同曝光规模比较。样本不足时少选或不做“爆款/低表现”结论，而不是用任意阈值补齐。

## 10. `caption_extract`

对已验证公开视频 URL 的代表视频调用 `caption_extract`（工具可用时），并将成功/失败写入账本。提取失败时，只分析实际可见的视频描述、标题、封面或页面文字，并明确“未取得字幕”；不得编造 Hook、口播、结构或 CTA。

## 11. 指标计算

仅在实际返回字段、统计窗口和样本定义足够时计算，并标出计算范围。

- 发布节奏：按可验证发布时间计算每周/每月视频数；样本不是完整时间窗时写为“样本内节奏”。
- 互动率：仅在播放量存在且大于零时计算：`(likes + comments + shares) / views`。缺任一互动字段时，改用已知分子并明确公式，或不计算。
- 内容支柱占比：仅按已分类且可见的样本视频计算，并报告未分类数。
- 商品、销量与销售额：分别展示字段和窗口，绝不互相替代。

绝不比较由不兼容分母计算的比率，例如一组按播放量、另一组按粉丝数，或不同窗口、不同来源口径的比率。

## 12. 缺失数据规则

- 未返回：写“未提供”；公开页面不可见：写“未公开”；请求或读取失败：写“读取失败”。
- 不把缺失值按零处理，不猜测字段，不用固定行业基准替代实时数据。
- 缺少影响主结论的信息时执行完整性门槛的一问一答；不影响主结论时继续并披露。
- 将缺失字段、结论影响、已尝试的补救和下一步所需信息写入报告。

## 13. 安全

不绕过登录、验证码、反爬、地域限制或访问控制；不要求或记录 API Key、Token、Cookie；不执行发布、私信、下单、店铺修改、广告投放等写操作。仅分析用户提供或公开可访问的信息，并避免将个人联系信息纳入报告。

## 14. 停止条件

在达到用户视频上限或默认 50 条、没有下一页、到达日期窗口边界、工具额度/限流、服务端停止、浏览器访问受限，或取得足以支持范围内结论的样本后停止。记录停止原因和完整性。若身份无法确认或关键缺失会改变主结论，停止深度分析并只问一个关键问题；用户要求继续时输出带限制的部分报告。
