# TikTok 账号分析与类目运营 Skills 设计

**日期：** 2026-07-20  
**状态：** 已在对话中确认，待实施  
**仓库：** `aronhy/tiktok-agent-skills`

## 1. 目标

在现有 `tiktok-shop-operator` 基础上新增三个相互独立、可以串联使用的 Skill：

1. `tiktok-account-audit`：输入任意公开 TikTok 账号链接，读取账号和视频，自动识别账号类型并生成运营诊断报告。
2. `tiktok-category-strategy`：输入目标类目与市场，结合商品、店铺、视频、达人和 TikTok 官方趋势信息生成类目运营方案。
3. `tiktok-growth-plan`：组合账号诊断和类目研究，输出账号进入目标类目的差距分析及 30 天行动计划。

所有报告必须结果前置、来源可追溯、缺失信息可见；信息不足且会改变结论时，AI 必须主动反问，不能编造数据。

## 2. 非目标

- 不绕过 TikTok 登录、验证码、反爬或访问控制。
- 不保证取得平台未公开的账号数据。
- 不执行发布视频、私信达人、修改店铺或投放广告等外部写操作。
- 不在 Skill 或示例配置中保存 API Key、Token 或其他真实凭证。
- 不用未经验证的固定行业基准替代实时数据。

## 3. 调研结论与数据边界

- TikTok Display API 可读取已授权用户的资料和近期公开视频，不适合直接分析任意竞争对手账号。
- TikTok Research API 可按用户名查询公开账号和视频，但需要研究项目审批，普通商业用户不能依赖它作为开箱即用的数据源。
- TikTok Creative Center 可提供按地区和行业筛选的热门产品、广告、关键词、视频、达人与趋势，适合作为类目研究的官方补充来源。
- 当前 KSS MCP 已有 `product_search`、`shop_search`、`video_search`、`creator_search` 和 `caption_extract`，但缺少按任意账号稳定读取资料及视频列表的明确接口。

因此本设计采用 KSS MCP 新工具作为主链路，公开网页读取作为降级链路。

## 4. 总体架构

```text
用户输入
  ↓
输入完整性检查
  ↓
MCP 主链路
  ├─ creator_profile
  └─ creator_videos
  ↓ 失败、缺失或接口不可用
浏览器公开页面降级
  ↓
数据完整性检查
  ↓ 关键字段仍不足
一次一个关键问题地向用户反问
  ↓
账号诊断 / 类目研究 / 增长计划
```

### 4.1 主链路接口

Skill 预期使用以下两个 MCP 工具；具体参数以 MCP 服务器实际暴露的 schema 为准，Skill 不得臆造未提供的参数。

#### `creator_profile`

输入优先级：`profileUrl` → `handle`。  
期望字段包括：达人 ID、Handle、昵称、简介、头像、地区、认证状态、粉丝数、关注数、累计获赞、视频数、账号链接，以及可用的店铺或商品关联信息。

#### `creator_videos`

输入优先级：达人 ID → Handle。  
默认目标是最近 50 条公开视频；若接口采用分页，应持续翻页直到达到目标数量、没有下一页、额度不足或服务端停止。期望字段包括：视频 ID、链接、标题或描述、发布时间、时长、播放、点赞、评论、分享、标签、达人信息，以及可用的商品、店铺、销量和销售额字段。

### 4.2 浏览器降级

当工具不存在、MCP 未连接、请求失败、结果为空或关键字段不足时，Skill 使用浏览器读取公开账号主页和公开的视频页面。浏览器只读取页面可见信息；遇到登录、验证码或访问限制时停止，并说明限制。

降级读取仍无法取得的字段统一标为“未公开”或“读取失败”，不将缺失值按零处理，不根据相邻字段猜测。

### 4.3 数据可信度

- A：账号资料和视频列表均来自 MCP，关键字段覆盖充分。
- B：MCP 与浏览器混合，或部分关键字段缺失。
- C：仅来自浏览器公开页面，数据覆盖有限。

每份报告列出来源、样本数量、覆盖时间、缺失字段、使用的假设和可信度等级。

## 5. 信息完整性与反问机制

执行任何深度分析前先检查：

1. 用户输入是否包含任务所需的账号链接、目标类目、国家或地区和运营目标。
2. MCP 或浏览器结果是否覆盖支撑结论的关键字段。
3. 数据缺失是否会改变账号分类、类目判断或行动建议。

处理规则：

- 能通过规范化输入、重试、分页、跨工具关联或浏览器降级取得的信息，先自行获取。
- 缺少的信息会改变主要结论时，必须暂停分析并向用户提问。
- 一次只问一个最关键的问题，并说明为什么需要它。
- 可选信息不影响主要结论时，使用明确标注的默认值继续。
- 用户明确要求在信息不足时继续，则输出部分报告，并标注缺失字段、影响范围和可信度。
- 不要求用户在对话中粘贴 API Key 或 Token；只提示其在本地 MCP 配置中完成设置。

## 6. Skill 设计

### 6.1 `tiktok-account-audit`

#### 触发条件

用户提供 TikTok 账号或主页链接，并要求分析账号、竞品、内容表现、带货表现、视频策略、运营问题或增长机会。

#### 输入

- 必需：公开 TikTok 账号链接。
- 可选：目标国家、分析周期、最多视频数、运营目标、对标账号。
- 默认：最多分析最近 50 条可取得的公开视频；不把默认值伪装成用户条件。

#### 账号类型识别

根据简介、链接、内容主题、商品关联、带货视频占比、店铺信息和品牌表达，将账号识别为内容型、带货型、品牌型或混合型。允许主类型加次类型；证据不足时标为“暂无法判断”并反问。

#### 工作流

1. 解析并规范化账号链接，提取 Handle。
2. 调用 `creator_profile` 和 `creator_videos`。
3. 必要时切换浏览器降级，并记录来源。
4. 检查完整性；必要时反问。
5. 计算发布节奏、互动表现和视频分布；只使用实际返回的字段。
6. 按播放、互动、销售额和近期增长选择代表视频。
7. 对代表视频使用 `caption_extract`；失败时分析可见描述并标注限制。
8. 识别内容支柱、Hook、结构、人设、卖点、CTA 和可复用模式。
9. 输出结果前置的运营诊断报告。

#### 输出

1. 一句话诊断。
2. 账号类型和当前阶段。
3. 三个应放大的优势。
4. 三个应优先解决的问题。
5. 账号数据、样本范围和可信度。
6. 发布节奏和内容支柱。
7. 爆款与低表现视频对比。
8. Hook、内容结构、人设、CTA 与字幕模式。
9. 可用时展示商品和带货表现。
10. 可复用内容公式。
11. 未来 7 天改进动作。

### 6.2 `tiktok-category-strategy`

#### 触发条件

用户希望选择、评估或运营某个 TikTok/TikTok Shop 类目，需要类目机会、竞争、商品、内容、达人或进入策略。

#### 输入

- 必需：目标类目、国家或地区。
- 可选：价格带、账号类型、商业目标、预算、内容语言、计划周期。
- 目标地区缺失时必须询问，因为趋势、价格、政策和竞争格局均受地区影响。

#### 工作流

1. 规范化类目名称和目标市场。
2. 使用 `product_search` 建立商品候选池。
3. 使用 `shop_search` 分析代表店铺与竞争强度。
4. 使用 `video_search` 查找类目或商品关联的高表现内容。
5. 使用 `creator_search` 建立达人地图。
6. 对代表视频使用 `caption_extract` 拆解内容。
7. 使用 TikTok Creative Center 的地区与行业趋势作为官方补充；不可访问时明确说明。
8. 交叉验证商品、店铺、视频、达人和趋势信号，不用单一爆款代表整个类目。
9. 输出结果前置的类目运营方案。

#### 输出

1. 类目是否值得进入，以及结论适用条件。
2. 最优切入方向。
3. 热门商品、价格带和需求信号。
4. 代表店铺和竞争强度。
5. 爆款视频的内容模式。
6. 适合合作或模仿的达人类型与样本。
7. 用户痛点和购买理由。
8. 内容支柱与选题矩阵。
9. 差异化定位。
10. 30 天类目运营方案。
11. 政策、供应、数据与执行风险。

### 6.3 `tiktok-growth-plan`

#### 触发条件

用户同时提供账号与目标类目，希望判断适配性、制定转型或增长路线，并获得可以直接执行的计划。

#### 输入

- 必需：TikTok 账号链接、目标类目、目标国家或地区、商业目标。
- 可选：预算、团队能力、库存情况、每日可投入时间、对标账号。

#### 工作流

1. 复用 `tiktok-account-audit` 的账号诊断流程。
2. 复用 `tiktok-category-strategy` 的类目研究流程。
3. 将账号能力与类目机会进行证据化匹配。
4. 区分可以保留的资产、需要停止的动作和需要补充的能力。
5. 从选品、内容、达人合作三条路径制定增长策略。
6. 输出 30 天行动日历、指标和下一轮 AI 任务。

#### 输出

1. 一句话运营定位。
2. 是否建议进入目标类目。
3. 账号与类目的适配度及证据。
4. 应保留、停止和新增的内容。
5. 选品、视频和达人三条增长路径。
6. 每周内容数量与内容比例。
7. 30 天行动日历。
8. 关键指标、复盘频率和调整条件。
9. 可直接复制给 AI 的下一步任务提示词。

## 7. 结果前置与报告规则

所有 Skill 的正式报告遵循相同顺序：

1. 结论。
2. 优先级行动。
3. 支撑结论的关键证据。
4. 完整分析和明细数据。
5. 数据限制与风险。

报告不得混淆播放量、销量和销售额；不得混淆视频发布时间与统计窗口；所有构造的 TikTok 链接必须有足够的 Handle 和 ID 支撑。

## 9. 验证场景

实施前先用不加载新 Skill 的代理执行基线场景，记录其是否遗漏降级、是否编造数据或是否跳过反问；实施后用相同场景复测。

至少覆盖：

1. 任意公开账号链接，MCP 两个新工具均可用。
2. `creator_profile` 可用但 `creator_videos` 不可用，需要浏览器降级。
3. MCP 完全未连接，浏览器只能取得部分公开视频。
4. 用户只说“我想做美妆”，缺少目标国家，必须反问。
5. 用户要求在信息不足时继续，必须输出部分报告和可信度。
6. 混合型账号同时包含普通内容和带货视频，不能只按单一类型评价。
7. 账号与目标类目明显不匹配，增长计划必须给出证据和替代路径。
8. 用户在对话中粘贴密钥，Skill 不得回显或写入仓库。

## 10. 验收标准

- 三个 Skill 均能被明确的自然语言请求触发，且不会互相抢占不相关任务。
- 账号分析优先调用 `creator_profile`、`creator_videos`，失败后才使用浏览器降级。
- 信息不足且影响结论时，AI 一次只问一个关键问题。
- 所有输出结果前置，并包含来源、覆盖范围、缺失信息和可信度。
- `SKILL.md` 与 `agents/openai.yaml` 一致并通过官方快速验证脚本。
- README 清晰列出新 Skill、适用场景、安装方法和示例。
- 仓库不包含真实凭证、缓存、临时文件或与项目无关的内容。
